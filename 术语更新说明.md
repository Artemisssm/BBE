# 术语更新说明

## 更新概述

为了避免术语混淆，提高系统专业性，我们将原来的**"单元名称"**统一改为**"链路模式"**。

## 术语对比

| 原术语 | 新术语 | 英文 | 说明 |
|--------|--------|------|------|
| 单元名称 | **链路模式** | Link Mode | 8种工作模式 |
| 单元类型 | 单元类型 | Unit Type | 4种功能模块（编码/调制/解调/译码） |

## 为什么要改？

### 问题1：术语混淆
- **单元名称**：返向中低速数传、返向高速数传...（8种）
- **单元类型**：编码、调制、解调、译码（4种）

两个都有"单元"，容易混淆！

### 问题2：不够专业
"单元名称"不是标准的通信术语，"链路模式"更专业。

### 问题3：理解困难
新用户不容易理解"单元名称"和"单元类型"的区别。

## 新术语的优势

### 1. 清晰明确
- **链路模式**：一看就知道是链路的工作模式
- **单元类型**：明确是功能模块的类型

### 2. 符合行业习惯
- 链路模式 (Link Mode) 是通信领域的标准术语
- 更容易被专业人员理解

### 3. 避免歧义
- "链路"和"单元"是不同层次的概念
- 不会再混淆

## 更新内容

### 1. 前端界面

#### 约束管理界面
```
旧：源单元名称 / 目标单元名称
新：源链路模式 / 目标链路模式
```

#### 表格列名
```
旧：源单元 / 目标单元
新：源链路 / 目标链路
```

#### 提示信息
```
旧：通常与源单元名称相同（约束发生在同一单元内）
新：通常与源链路模式相同（约束发生在同一链路内）
```

#### 变量名
```javascript
旧：unitNameOptions
新：linkModeOptions
```

### 2. 数据库注释

```sql
-- 旧注释
source_unit_name VARCHAR(50) COMMENT '源单元名称（*表示通配）'
target_unit_name VARCHAR(50) COMMENT '目标单元名称（*表示通配）'

-- 新注释
source_unit_name VARCHAR(50) COMMENT '源链路模式（*表示通配）'
target_unit_name VARCHAR(50) COMMENT '目标链路模式（*表示通配）'
```

**注意**：字段名保持不变（`source_unit_name`），只更新注释。

### 3. 文档

所有文档中的"单元名称"都改为"链路模式"。

## 8种链路模式

1. **返向中低速数传** - 返回链路，中低速数据传输
2. **返向高速数传** - 返回链路，高速数据传输
3. **ACM数传** - 自适应编码调制数据传输
4. **前向数传小环** - 前向链路，小环回测试
5. **低速模拟源** - 低速模拟信号源
6. **高速模拟源** - 高速模拟信号源
7. **ACM模拟源** - ACM模式模拟信号源
8. **前向数传** - 前向链路数据传输

## 4种单元类型

1. **编码单元** (ENCODE) - 信道编码
2. **调制单元** (MODULATE) - 信号调制
3. **解调单元** (DEMODULATE) - 信号解调
4. **译码单元** (DECODE) - 信道译码

## 层次关系

```
链路模式（8种）
  └─ 单元类型（4种）
      └─ 参数（多个）
```

**示例**：
```
返向中低速数传（链路模式）
  ├─ 编码单元（单元类型）
  │   ├─ 编码方式（参数）
  │   └─ 编码码率（参数）
  ├─ 调制单元（单元类型）
  │   ├─ 调制方式（参数）
  │   └─ 符号率（参数）
  └─ ...
```

## 约束关系说明

### 同链路、同单元类型
```
链路模式: 返向中低速数传
源单元类型: 调制单元
源参数: 调制方式
↓
目标单元类型: 调制单元 (相同)
目标参数: 符号率
```

### 同链路、跨单元类型
```
链路模式: ACM数传
源单元类型: 编码单元
源参数: ACM使能
↓
目标单元类型: 调制单元 (不同)
目标参数: ACM使能
```

### 通配链路模式
```
链路模式: * (所有链路)
源单元类型: 编码单元
源参数: 编码方式
↓
目标单元类型: 编码单元
目标参数: 编码码率
```
